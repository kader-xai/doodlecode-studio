# File format specification

DoodleCode Studio reads and writes plain `.py` files. Cell boundaries and
callouts live in **comments** so the file remains a valid Python script
that runs end-to-end (or imports cleanly) outside the app.

This document is the **stable** part of the API. Anything described here
is guaranteed to round-trip and is **backward-compatible** between
format versions.

## Format versions

| Version | Introduced in app | Adds |
|---------|-------------------|------|
| **v1**  | `0.1.0`           | `# %% kind=… color=… title="…"` headers, `# @explain:` / `# @title:` / `# @kind:` / `# @color:` / `# @tags:` directives. |
| **v2**  | `0.2.0`           | `# @callout` separator for multiple callouts per cell, `# @image:` attachment (data URL or path). |

The serializer writes a header comment as the first line:

```text
# doodlecode format-version: 2
# Generated by DoodleCode Studio. Safe to edit; see docs/FILE_FORMAT.md.
```

v1 files have no such header (`version` is assumed to be `1`). The v2
parser reads both. **Old readers don't break on v2 files** — extra
callouts use `# @callout` lines that look like ordinary comments and are
ignored by the v1 directive scanner.

## At a glance

```python
# %% [markdown]
# # A heading
# Some descriptive text in **markdown**.

# %% kind=function color=mint title="What this section does"
# @explain: One sentence per @explain line. Multiple lines are joined
# @explain: with a single space and become the bubble body.
# @tags: math, geometry
import math

def area_of_circle(r):
    return math.pi * r * r
```

## Cell delimiters

A new cell starts on **any line** matching the regex `^\s*#\s*%%\s*(.*)$`.
Everything between two markers is the cell's body. Content before the
first marker (if any) becomes a leading code cell.

The captured group after `# %%` is the **header**. Three shapes:

| Header                          | Cell kind        |
|---------------------------------|------------------|
| `[markdown]` or `markdown`      | Markdown cell    |
| `kind=… color=… title="…"`      | Code cell + meta |
| *(empty)*                       | Plain code cell  |

The header is whitespace-tolerant; case is preserved for values.

## Markdown cells

A markdown cell's body is comprised of comment lines. Each line gets one
leading `#` (optionally followed by a space) stripped on read. Blank
lines must be `#` or empty. Example:

```python
# %% [markdown]
# ## Section title
#
# - bullet one
# - bullet two
```

Renders as a doodle text card with headings, bullets, **bold**, and
`inline code`.

## Code cells

A code cell's header may carry up to four `key=value` pairs:

| Key      | Type   | Purpose |
|----------|--------|---------|
| `kind`   | string | Free-text label (e.g. `function`, `loop`, `data`). Used to pick a fallback color and shown as a chip. |
| `color`  | string | Named color from the [palette](#palette). Overrides the kind-based default. |
| `title`  | string | Short title shown in big handwriting. Quote if it contains spaces (`title="My demo"`). |
| `tags`   | string | Comma-separated. Cosmetic / future use. |

**Quoting.** Use double quotes around values that contain spaces or
quotes. Backslash-escapes work (`\"`, `\\`).

### `# @` directives

Lines starting with `# @<key>: <value>` immediately after the header (with
optional blank lines between them) become cell metadata, and are
**stripped from the source** the editor sees.

| Directive    | Since | Behavior |
|--------------|-------|----------|
| `# @title:`  | v1    | Sets / overrides the title from the header. |
| `# @explain:`| v1    | One line of the callout body. Multiple lines are joined with single spaces. |
| `# @kind:`   | v1    | Sets / overrides `kind`. |
| `# @color:`  | v1    | Sets / overrides `color`. |
| `# @tags:`   | v1    | Comma-separated tags. |
| `# @image:`  | v2    | Data URL or path. Renders as a thumbnail inside the bubble. Keep under ~1.5 MB. |
| `# @callout` | v2    | Opens a **new** callout for the same cell. May carry inline `title=` / `color=` / `kind=` like the `# %%` header. |

Directives end at the first non-comment, non-directive line. From there
on the cell is pure code.

Example — same callout, three ways:

```python
# header-only
# %% kind=function color=mint title="Compute area" explain="Returns π·r²"
def f(r): return 3.14 * r * r

# directives-only (preferred for multi-line bodies)
# %%
# @title: Compute area
# @kind: function
# @color: mint
# @explain: Returns π·r². Useful for circles.
def f(r): return 3.14 * r * r

# header + directives (the default produced by 💾 Save)
# %% kind=function color=mint title="Compute area"
# @explain: Returns π·r². Useful for circles.
def f(r): return 3.14 * r * r
```

## Palette

Named colors usable in `color=` (case-sensitive):

| Name       | Light hex | Dark hex |
|------------|-----------|----------|
| `yellow`   | `#ffec99` | `#a37f08` |
| `pink`     | `#fcc2d7` | `#a61e4d` |
| `mint`     | `#b2f2bb` | `#2b8a3e` |
| `sky`      | `#a5d8ff` | `#1864ab` |
| `peach`    | `#ffd8a8` | `#c2410c` |
| `lavender` | `#d0bfff` | `#6741d9` |
| `rose`     | `#ffc9c9` | `#c92a2a` |
| `cyan`     | `#99e9f2` | `#0c8599` |
| `violet`   | `#eebefa` | `#862e9c` |
| `green`    | `#8ce99a` | `#2f9e44` |
| `blue`     | `#74c0fc` | `#1971c2` |
| `orange`   | `#ffa94d` | `#d9480f` |
| `red`      | `#ff8787` | `#c92a2a` |
| `paper`    | `#fdf7e6` | `#2a2418` |
| `white`    | `#ffffff` | `#1a1a1a` |

If `color` is omitted, a default is picked from `kind` via this map:

```
import → sky      function → mint    class → pink
loop / conditional → yellow          assign → peach
return → lavender                    try → rose
context / intro → sky                install → rose
```

Raw hex codes are also accepted in `color=#abcdef` form — they're passed
through unchanged.

## Multiple callouts per cell (v2)

```python
# %% kind=function color=mint title="Train loop"
# @explain: Repeat 100 times: predict → loss → backprop → step.
# @image: data:image/png;base64,iVBORw0KGgo…
# @callout title="Aside · learning rate" color=peach
# @explain: lr=0.01 is the step size. Smaller is more stable, slower.
# @callout color=lavender
# @title: When to stop
# @explain: Watch the validation loss, not the training loss.
for epoch in range(100):
    ...
```

- The **first** callout's `kind` / `color` / `title` ride on the `# %%`
  header for v1 readability.
- Each `# @callout` starts a new bubble. Use the inline form for short
  attributes or follow up with `# @title:` / `# @color:` lines.
- Order is preserved on save.

## Source body

Everything after the directives, up to the next `# %%` marker (or end of
file), is the cell's executable source. It is preserved **byte for byte**
on round-trip apart from trailing whitespace normalization.

There is no syntax check performed at upload time — the file can have
syntax errors and the cells will still split correctly.

## Round-trip guarantees

These hold from `0.1.0`:

1. **Upload then Save** yields a file whose markers, meta fields, and
   source bodies match the original, up to comment-formatting
   normalisation (e.g. `# %%` may gain a trailing space).
2. Cells with no meta serialize without `kind=`/`color=`/`title=` keys.
3. Directives serialize in stable order: `@title`, `@kind`, `@color`,
   `@explain`, `@tags`.
4. Long `@explain` bodies are wrapped at ~72 columns on save.

What is **not** guaranteed:

- Comment style inside cell bodies — your `#` comments are preserved
  verbatim, but blank-line counts between cells may change.
- Field order inside a header line may change (only directive order is
  stable).

## Compatibility with VS Code & Jupytext

The `# %%` cell-marker convention is shared with VS Code's "Interactive
Window" and `jupytext`'s "percent" format, so DoodleCode files open
cleanly in both. The `# @explain:` directives are unique to DoodleCode
and will be visible as comments — non-disruptive but uninterpreted.

## Schema (JSON, internal)

The HTTP API uses this shape (Pydantic):

```python
class CellMeta(BaseModel):
    kind: Optional[str]
    color: Optional[str]
    title: Optional[str]
    explain: Optional[str]
    tags: list[str]

class Cell(BaseModel):
    id: str
    kind: Literal["code", "markdown"]
    source: str
    meta: Optional[CellMeta]

class Notebook(BaseModel):
    name: str
    cells: list[Cell]
```

`POST /upload` returns a `Notebook`; `POST /export` accepts one and
returns the serialized `.py` text.
